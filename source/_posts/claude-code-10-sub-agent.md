当任务变复杂时，一个 Claude Code 会话可能不够用。

这时就需要 Sub Agent，也就是子 Agent。

你可以把它理解成 Claude Code 派出去的分身或下属。它们有自己的上下文，可以并行干活，最后把结果汇总给主 Agent。

## 为什么需要 Sub Agent？

假设你正在做一个番茄钟应用，想上线前调研竞品。

这个任务可能包含：

- 调研其他番茄钟产品功能
- 分析它们的 UI 设计
- 查看用户评价
- 总结商业化方式
- 给当前项目提出建议

如果主 Agent 一件件串行做，问题很明显：

- 耗时长
- 主上下文塞满调研细节
- 写代码任务被打断
- 后续开发时还背着大量无关信息

更好的方式是派子 Agent 分头做调研。

## Sub Agent 的核心价值

Sub Agent 的优势是：

- 有独立上下文
- 可以并行执行
- 不污染主 Agent 的上下文
- 适合调研、审查、搜索、对比、总结

主 Agent 不需要记住所有调研细节，只需要拿到结论。

这对长任务非常有用。

## 适合派 Sub Agent 的任务

Sub Agent 特别适合这些任务：

- 多方向调研
- 多文件代码审查
- 多方案比较
- 搜索资料并总结
- 竞品分析
- 测试失败原因排查
- 安全审查
- 文档整理

如果任务本身非常简单，就没必要派子 Agent。

## 示例：让多个子 Agent 做调研

你可以这样说：

```text
请派生多个子 Agent，分别调研竞品功能、UI 设计、用户评价和商业化方式，最后汇总成一份开发建议。不要修改当前项目文件。
```

这个提示词明确了：

- 要派生多个子 Agent
- 每个子 Agent 的方向
- 最终输出形式
- 不允许修改文件

这会比一句“帮我调研一下竞品”靠谱很多。

## 示例：让子 Agent 做代码审查

你也可以让子 Agent 审查不同角度：

```text
请派生 3 个子 Agent 审查当前改动。一个看代码质量，一个看性能问题，一个看可维护性。最后按严重程度汇总问题，不要直接修改文件。
```

这类任务很适合并行。

因为不同审查角度可以互不干扰。

## 手动创建 Sub Agent

有些版本或环境里，你可以通过类似 `/agents` 的入口创建自定义 Agent。

创建时通常需要说明：

- Agent 名称
- 它擅长什么
- 什么时候调用
- 可以使用哪些工具
- 输出格式要求

比如你可以创建一个“内容审校 Agent”，专门检查文章逻辑、错别字和结构。

也可以创建一个“前端审查 Agent”，专门检查 UI、交互和响应式问题。

## 使用 Sub Agent 的注意事项

Sub Agent 不是越多越好。

派太多会带来：

- 成本增加
- 结果重复
- 汇总困难
- 任务边界混乱

好的子 Agent 任务应该边界清晰。

比如“你只负责调研 UI 设计趋势，不要分析商业化，也不要写代码”。

## 主 Agent 和 Sub Agent 的分工

一个好的工作方式是：

- 主 Agent 负责目标、约束、汇总和决策
- Sub Agent 负责独立搜索、审查、分析和报告

主 Agent 不应该把所有细节都背下来。

它只需要最终结论、证据和建议。

这也是 Sub Agent 最重要的意义：让复杂任务分而治之。
