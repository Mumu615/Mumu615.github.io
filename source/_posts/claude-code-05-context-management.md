Claude Code 用久了，你可能会发现它变慢、变贵、变笨，甚至开始忘记重点。

这不是玄学，而是上下文窗口的问题。

这一篇讲两个核心能力：**上下文管理** 和 **风险回滚**。

## 为什么 Claude Code 会变慢、变笨？

所有大模型都有上下文窗口限制。

你和 Claude Code 说过的话、它读过的文件、运行命令的输出、报错日志、修改计划，都会占用上下文。

上下文越来越长后，会出现几个问题：

- 回复变慢
- Token 消耗变高
- 模型抓不住重点
- 对旧信息过度依赖
- 修改质量下降

即使模型宣称支持很长上下文，有效上下文也不是无限的。

所以你需要主动管理上下文。

## 阶段性使用 /compact

当你完成一个阶段，比如：

- 项目初始化完成
- 页面第一版完成
- Bug 修复完成
- 调研阶段完成

就可以执行：

```text
/compact
```

让 Claude Code 压缩上下文。

这能让它在后续任务里更专注。

你可以把 `/compact` 理解成整理会议纪要：过程被压缩，结论被留下。

## 必要时使用 /clear

如果当前对话已经混乱，或者你要切换到完全不同的任务，就用：

```text
/clear
```

这相当于重新开一局。

但不要把 `/clear` 当成万能清洁剂。

如果当前任务还没完成，或者很多重要上下文没有沉淀到文件里，直接清空可能会让 Claude Code 丢失关键背景。

更稳的做法是：

1. 先让它总结当前进度
2. 把总结写进项目文档
3. 再 `/clear`
4. 新会话从文档恢复上下文

## 查看上下文占用

你可以用：

```text
/context
```

查看当前上下文情况。

如果占用已经很高，或者你明显感觉它开始答非所问，就该压缩了。

我的建议是：不要等到系统自动提醒上下文快满。完成一个阶段，就主动整理。

## 继续旧会话

如果你退出了 Claude Code，之后想恢复之前的会话，可以使用恢复相关命令。

有些版本支持在 Claude Code 内选择恢复历史会话，也可以用 continue 方式启动。

核心思路是：不要把一个会话无限聊下去，也不要完全不保存阶段成果。

长期项目最好配合 Git、文档和 `CLAUDE.md` 管理。

## 回滚：Claude Code 自带撤回不够

Claude Code 有自带回滚功能。

你可以通过 `/rewind` 或快捷回滚界面撤回：

- 仅回滚对话
- 回滚对话和文件修改
- 仅回滚文件修改

但它有一个重要限制：**它只能回滚 Claude Code 自己编辑过的文件。**

如果它执行终端命令下载了依赖、生成了文件、修改了锁文件，或者你自己手动改了内容，它不一定能完整撤回。

所以真正靠谱的后悔药是 Git。

## 把 Git 理解成游戏存档

Git 对小白来说，不需要一开始理解复杂命令。

你只要先记住这个比喻：**Git 是项目的游戏存档系统。**

每完成一个阶段，就存一次档。

改坏了，就回到上一个档。

你可以让 Claude Code 帮你做 Git 操作，比如：

```text
请帮我检查当前项目是否已经初始化 Git。如果没有，请初始化 Git，并提交当前可运行版本作为第一个存档。
```

后续你可以经常说：

```text
现在这个版本我满意，请帮我提交一个 Git 存档。
```

或者：

```text
刚才改坏了，请帮我回到上一个可用版本。先告诉我会回滚哪些文件，再执行。
```

有了 Git 兜底，你才敢让 Claude Code 大胆尝试。

## 推荐工作节奏

日常使用时，可以按这个节奏：

1. 开始任务前先写清楚目标
2. 让 Claude Code 先给计划
3. 同意后执行
4. 完成一个阶段后运行检查
5. 版本满意就 Git 提交
6. 阶段结束后 `/compact`
7. 长任务必要时写阶段总结文档

这套流程会显著降低失控风险。
