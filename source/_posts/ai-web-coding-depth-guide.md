好久不见！大家好，我是 AI 狐小乐（Junck）。

人类正在加速跨入 AI 时代，各种大模型和工具层出不穷。普通人和“超级个体”之间的差距，往往就产生于对 AI 的理解和应用中。最近这一段时间，我把主要的精力都投入到了 AI Web Coding 的实践上。

今天就和大家详细聊聊我近期的实战体验。文章主要分为两个部分：我个人的核心体验总结，以及一套从 0 到 1 的高效工作流分享。希望能帮大家少走弯路！

## 一、核心爽点与现实挑战

在深入体验了各种 AI 编码工具后，我的整体感受是：ROI（投资回报率）已经完全可以接受，是时候把它接入你的日常开发中了。

### 1）绝对的“爽点”：生产力跃迁

最大的爽点非常明确——效率的呈指数级提升。

现在的开发状态是：我可以同时开启多个窗口让 AI 并发 Coding。我不再是一个纯粹的“码农”，而是变成了一个**“代行人”或“项目经理”**。AI 就像一个技术极强但需要引导的新人，它负责输出代码和申请权限，我只负责计划（Plan）和检查（Review）。

### 2）必须正视的“挑战”

当然，想要用得舒心，目前还存在几个明显的门槛：

- **对开发者的架构能力要求极高**：你前期的系统架构设计得越清晰，AI 生成代码的效果就越好，耗时也越短。
- **测试与集成能力是核心**：自动化 Coding 的关键在于让 AI 具备自我验证和自我纠错的能力。这极其依赖测试用例（Test Case）的覆盖率。覆盖率足够高，人工介入就少；否则你还得不断帮它找 Bug。
- **专注度与权限管理的博弈**：AI 在长达 20 分钟甚至 1 小时的编码过程中，会时不时打断你，要求运行权限或人工 Review。
  - **实战建议**：出于安全考虑，不建议直接给满宿主机权限。更好的做法是把 AI 丢进虚拟机（VM）里，给足权限让它自己折腾，这样既不打断你的思路，也不用担心它搞坏环境。
- **精力分配与成本（贵！）**：计划和检查的工作量其实并不小（尤其是前期的规划）。此外，目前好用的工具确实比较昂贵（Expensive），但好在能换来实打实的产出。

## 二、我的 0 到 1 实战工作流（Workflow）

结合我在构建 [Aether AI](https://www.google.com/search?q=%23) 等项目时的经验，我总结了一套调用各类大模型接口的高效工作流，主要借助 Gemini、Claude Code 以及轻量级框架 OpenSpec 来完成。

### Step 1：需求与方案设计（双 AI 交叉验证）

不要直接让 AI 开始写代码，先做设计！

1. **初始对话**：找代码能力极强的 Gemini 聊需求、方案和接口设计，让它输出一份初步的需求文档（PRD）和架构划分。
2. **交叉验证（Double Check）**：将 Gemini 聊出来的记录或文档，扔给另一个大模型（比如 Claude），设定 Prompt：“我朋友做了个设计，你从你的专业角度帮忙 review 一下”。通过不同 AI 的视角查漏补缺，此时的文档就基本可用了。

### Step 2：初始化 OpenSpec 与项目基准

为了让 AI 拥有持续的“记忆”并且脱离杂乱的聊天上下文，我们需要引入 OpenSpec。

- **OpenSpec 是什么？**
  它可以理解为一个轻量级的规范框架，配合 Codex 或 Claude Code 使用。它通过特定的 Prompt 集合和落地的 Markdown 文件（如 Proposal、Task），让 AI 知道“我们在哪，我们要做什么，接下来该怎么做”。
- **操作**：初始化项目，填好 `project.md`。这里面主要写明你的项目要求、技术栈（比如 Vue 3、Go、Python 等）和测试路径。
  - **Tips**：各种工具的选择不用强求一致性，用着顺手、能 Work 就行。

### Step 3：生成 Proposal 与 Task（任务拆解）

把第一步确定的计划喂给初始化好的 AI 工具，让它生成详细的 Proposal（提案）和 Task（任务列表）。

- 在这个阶段你会和 AI 进行多轮确认，最终敲定它要分哪几个模块、拆成 1/2/3/4/5 几个步骤。
- **核心动作**：必须在这个阶段提供测试标准，告诉 AI 如何验证它写的代码，这是实现“全自动纠错”的灵魂。

### Step 4：Apply（自动执行与自我验证）

确认无误后，执行 Apply。AI 会根据 Task 逐步开始实现代码。

- 这个过程可能持续 20 分钟到 1 小时不等。如果你的 `project.md` 限制加得好（比如规定了明确的测试工具和路径），AI 就会自己默默干活，大大减少对你的打扰。
- 遇到跑偏的地方，及时给出反馈让它修改。

### Step 5：归档基准（Truth）与人工 Code Review

- **建立基准**：当一个功能确认实现后，一定要将其归档。这会变成你项目的已知事实（Truth）。后续做 1 到 N 的需求迭代时，有了这个基准，AI 才不会把原来好好的功能给改坏，保持代码的连续性。
- **人工 Review 的底线**：即使前期做了无数次 Double Check，最终依然要人眼去看 AI 写的 Spec、Task 和最终源码。
  - **避坑指南**：要特别警惕“Bug 叠 Bug”的情况——有时候测试用例通过了，不是因为代码写对了，而是测试代码和业务代码错到一块儿去了。这种时候就必须发挥人类开发者的兜底作用。

## 总结

AI Web Coding 的核心，不是“把写代码这件事彻底交给 AI”，而是你先把系统边界、质量标准和验证机制设好，再让 AI 作为高效率执行层去放大你的产能。

说白了：

- 架构能力，决定上限；
- 测试体系，决定稳定性；
- 人工 Review，决定最终质量。

当你把这三件事抓稳，AI 才会真正成为你的生产力引擎，而不是新的技术债制造机。
